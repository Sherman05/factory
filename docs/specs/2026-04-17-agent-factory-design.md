## 1. Цель

Построить персональную "фабрику" AI-агентов, которая максимально автоматизирует полный цикл разработки продуктов: постановка задачи → планирование → дизайн (если нужен UI) → код → тесты → ревью → merge → деплой. Человек (владелец) остаётся в цикле только как постановщик задач и финальный апрувер merge-ей — это осознанный выбор ради безопасности, а не техническое ограничение.

**Успех = владелец управляет фабрикой с телефона через Telegram-бот: бросает идеи, получает уведомления о готовых PR, апрувит merge одной кнопкой. Фабрика работает параллельно над 3-4 задачами.**

## 2. Scope

### В scope этого спека
- Базовая инфраструктура фабрики на vibe-kanban + Claude Code
- 5 специализированных ролей агентов (Planner, Designer, Coder, Tester, Reviewer)
- Первый продукт фабрики: Telegram Control Bot (будет и "пультом управления")
- Локальный запуск на ПК владельца (Windows 11)
- Git-первый workflow: всё через worktrees и PR

### Вне scope (явно)
- Облачный хостинг фабрики 24/7 (VPS, Kubernetes) — позже, если потребуется
- Многопользовательский режим / команды — это персональная фабрика
- Замена владельца в merge-решениях — человек всегда апрувит merge в main
- Неограниченная автономия на длинных задачах (>1 дня) — все задачи должны укладываться в ≤2 часа работы агента; большие задачи Planner разбивает

## 4. Роли агентов

Все роли — это `.claude/agents/<role>.md` файлы (Claude Code subagents). Каждая роль = шаблон системного промпта + whitelist skill'ов. Один Claude Code "перевоплощается" в нужную роль по задаче. Контекст не переносится между задачами — каждая задача это свежий spawn с чистым окном.

### 4.1 🧠 Planner
- **Вход:** высокоуровневая задача владельца ("сделай бот для заметок")
- **Выход:** документ с подзадачами (каждая ≤2 часа работы агента), критерии успеха, пайплайн
- **Skill'ы:** `superpowers:brainstorming`, `superpowers:writing-plans`
- **Не делает:** не пишет код, не делает дизайн

### 4.2 🎨 Designer
- **Вход:** подзадача с UI-компонентом от Planner
- **Выход:** дизайн-система (цвета, типографика), мокапы/вайрфреймы, скетчи в HTML
- **Skill'ы:** `frontend-design:frontend-design`, `frontend:tailwind-theme-builder`, `frontend:shadcn-ui`, `frontend:design-review`
- **Не делает:** не пишет бизнес-логику, не деплоит

### 4.3 👨‍💻 Coder
- **Вход:** подзадача + (если UI) готовый макет от Designer
- **Выход:** рабочий код в feature-ветке, коммиты с понятными сообщениями
- **Skill'ы:** `superpowers:test-driven-development`, `superpowers:systematic-debugging`, `frontend:react-patterns`, `cloudflare:cloudflare-worker-builder`, etc.
- **Правила (в CLAUDE.md):**
  - Сначала тест, потом код (TDD)
  - Не вводить зависимости без необходимости
  - Не писать комментарии "что делает код" (self-documenting)
  - Bug-fix = та же роль, вход: bug-report, выход: failing test → fix → passing test

### 4.4 🧪 Tester
- **Вход:** готовая feature-ветка от Coder
- **Выход:** дополнительные тесты (edge cases, error paths, integration), отчёт о покрытии
- **Skill'ы:** `superpowers:verification-before-completion`
- **Не делает:** не меняет продакшн-код (только тесты); если найдена проблема — заводит новую задачу Coder'у

### 4.5 🔍 Reviewer
- **Вход:** feature-ветка + тесты
- **Выход:** PR с диффом, комментариями, вердиктом (approve / request-changes)
- **Skill'ы:** `superpowers:requesting-code-review`, `superpowers:receiving-code-review`, `simplify`
- **Проверяет:** соответствие спеку, отсутствие YAGNI, безопасность (OWASP), отсутствие хардкода секретов, адекватность тестов

## 5. Пайплайны задач

### 5.1 Backend / API задача
```
Planner → Coder → Tester → Reviewer → 👤 владелец → merge → auto-deploy
```

### 5.2 UI задача
```
Planner → Designer → Coder → Tester → Reviewer → 👤 владелец → merge → auto-deploy
```

### 5.3 Bug-fix
```
Planner (мини: воспроизведение + критерий fix) → Coder (TDD: failing test → fix) → Reviewer → 👤 владелец → merge
```

### 5.4 Параллельность
- **Max 20x подписка** → **3–4 Claude Code параллельно** без тротлинга
- vibe-kanban следит за лимитами, очередь задач
- Каждая задача изолирована в своём git worktree — мержи не конфликтуют "на лету"

## 6. Первый продукт фабрики: Telegram Control Bot

**Зачем:** валидация, что фабрика работает + сразу получить "пульт управления с телефона".

**Функции v1:**
- Получает webhooks от vibe-kanban через MCP (или polling API) → шлёт уведомления владельцу в Telegram
- Инлайн-кнопки: `✅ Approve & Merge` / `❌ Reject` / `💬 Comment`
- Команда `/new <описание>` — создать задачу в vibe-kanban из чата
- Команда `/status` — текущая доска (1 картинка/текст)
- Whitelist по Telegram user_id (только владелец, никого больше)

**Технический стек (предварительно, Coder подтвердит):**
- Node.js + TypeScript + Telegraf (или grammY)
- Деплой: Cloudflare Workers (бесплатный tier, webhooks)
- Хранение: KV-store для связки Telegram user_id ↔ задачи

**Out of scope v1:** голосовой ввод, групповые чаты, права нескольких пользователей.

## 9. Критерии успеха (как поймём, что фабрика работает)

1. **MVP-критерий (минимальный срез Telegram Control Bot):** владелец бросает задачу "сделай бот, который получает webhook от vibe-kanban и шлёт уведомление мне в Telegram" → через ≤2 часа готовый PR, тесты прошли, после апрува — уведомление приходит в реальный чат
2. **Main deliverable (полный Telegram Control Bot):** владелец с телефона получает уведомление о готовом PR → апрувит inline-кнопкой → видит автодеплой. Может командой `/new <описание>` создать задачу в kanban
3. **Sustainable:** параллельно крутятся ≥2 задачи без конфликтов, Max 20x лимиты не упираются в потолок за день обычного использования

## 10. Риски и смягчение

| Риск                                                     | Смягчение                                                  |
|----------------------------------------------------------|------------------------------------------------------------|
| Агент накосячит в большой задаче (ошибки накапливаются)  | Planner разбивает задачи на ≤2 часа; короткие контексты    |
| Claude Max лимиты упрутся в потолок                      | Очередь в vibe-kanban; не спамить задачами; 3-4 параллельно |
| Агент закоммитит секреты в git                           | Hook pre-commit на `gitleaks` / секрет-сканер              |
| Reviewer "пропустит" плохой код                          | Владелец — последний рубеж; не убираем human-in-the-loop   |
| Telegram-бот упадёт → вы не получите уведомления         | Healthcheck в vibe-kanban; fallback — смотрите веб-UI       |
| Дрифт контекста в длинных задачах                         | Задачи разбиты; TDD-цикл заставляет агента фокусироваться  |
| Разработка фабрики затянется                              | Итеративно: сначала MVP (ручной запуск), потом автоматизация|

## 11. Будущие расширения (не сейчас)

- Облачный хостинг фабрики 24/7 (VPS + tailscale для доступа)
- Второй агент (Codex/Gemini) как "сравнительный ревьюер" (diff двух реализаций)
- Голосовой ввод задач через Telegram
- Автоматическое закрытие Issues в GitHub → задача в kanban
- Интеграция с Linear / Notion для постановки задач
- Метрики фабрики (время задачи, success rate, cost per task)
